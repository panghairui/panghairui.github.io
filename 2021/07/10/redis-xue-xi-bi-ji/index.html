一、前言

之前写的Redis学习笔记的博客，个人认为写的不是很好，不够系统。故打算从头重学Redis，整理学习笔记。

二、Redis初识

2.1 Redis是什么

  • 开源：可以看Redis的源码，去学习，企业可以做一些定制化。
  • 基于键值的存储服务系统
  • 多种数据结构
  • 高性能、功能丰富

2.2 谁在使用Redis

  • Github
  • Twitter
  • stackoverflow
  • 阿里巴巴
  • 百度
  • 新浪
  • 美团
  • 搜狐

只要是正常互联网企业，都在应用Redis。不会吧，不会吧，不会还有公司没用Redis吧。

2.3 Redis的特性

  • 速度快
  • 持久化
  • 多种数据结构
  • 支持多种编程语言
  • 功能丰富
  • 简单
  • 主从复制
  • 高可用、分布式

速度快

每秒10W次读写

为什么这么快呢？

Redis是将数据存到内存当中的，内存读取肯定比磁盘快。（主要原因）

而且Redis是基于C语言实现的，合理使用C语言，效率极高。

Redis早期版本使用单线程，避免切换上下文耗费时间。

Redis6.0版本后，使用多线程，主要是为了处理网络I/O的瓶颈。

持久化（断电不丢数据）

Redis所有的数据保存在内存中，对数据的更新将异步的保存到磁盘上。

数据结构

提供5种主要的数据结构

除了5种主要的数据结构，还有一些其他的数据结构，如BitMaps（位图），HyperLogLog（超小内存唯一值计数），GEO（地理信息定位）。

支持多种客户端语言

  • Java
  • php
  • python
  • Ruby
  • Lua
  • nodeJS

功能丰富

  • 发布订阅
  • Lua脚本
  • 事务
  • pipeline

简单

源代码可阅读化，可定制化。

不依赖外部库。

单线程模型。

主从复制

高可用、分布式

Redis-Sentinel(v2.8)支持高可用

Redis-Cluster(v3.0)支持分布式

2.4 Redis典型使用场景

  • 缓存系统
  • 计数器
  • 消息队列系统
  • 排行榜
  • 社交网络
  • 实时系统

三、API 的理解和使用

3.1 通用命令

  • keys 计算数据库里所有的健
  • dbsize 算出数据库的大小
  • exists key 判断一个 key 是否存在
  • del key [key …] 删除 key
  • expire key seconds 为 key 设置过期时间
  • type key 查看数据类型

keys 命令

API

keys * # 遍历所有key
keys [pattern] # 遍历合适的key

keys 命令一般不在生产环境使用，keys命令是一个O(N)的命令

那怎么用呢？ 热备从节点（之后会讲）

dpsize 命令

dpsize # 计算key的总数

时间复杂度是O(1)

exists 命令

exists key # 检查key是否存在

时间复杂度是O(1)

del 命令

del key # 删除指定的key-value

expire ttl presist 命令

expire key seconds # key在seconds秒后过期
ttl key # 查看key剩余的过期时间
persist key # 去掉key的过期时间

type 命令

type key # 返回key的类型

时间复杂度总览

3.2 数据结构和内部编码

Redis 是基于内存的数据库，而内存是相对来说比较昂贵的。

Redis 源码内部是有 redisObject 这样一个结构体。

这个结构体包含了 数据类型 编码方式 数据指针 虚拟内存 等等信息。

3.3 单线程

我们可以理解为，Redis 执行命令的时候，命令就进入了高速公路，只能一个一个来。

也就是说，Redis 在一个瞬间，只会执行一条命令。

单线程为什么这么快？

  • 纯内存（最主要原因）
  • 非阻塞 IO（不会在 IO 上浪费特别多的时间）

  • 避免线程切换和竞态消耗

单线程需要注意什么？

  • 一次只运行一条命令
  • 拒绝长（慢）命令
  • 其实不是单线程（执行 fysnc / close file descriptor 命令的时候会有额外的线程来做）

3.4 字符串

结构

value 可以存 字符串 数字 二进制 JSON

value 的大小限制是 512M

主要使用场景

  • 缓存
  • 计数器
  • 分布式锁

常用命令

get key # 获取 key 对应的 value
set key value # 设置 key-value
del key # 删除key-value

incr key # 自增1
decr key # 自减1
incrby key k # 自增k
decrby key k # 自减k

set key value # 不管 key 是否存在，都设置
setnx key value # key 不存在，才设置
set key value xx # key 存在，才设置

mget key1 key2 key3... # 批量获取 key，原子操作
mset key1 value1 key2 value2 # 批量设置 key-value

getset key newvalue # set key newvalue 并返回旧的 value
append key value # 将 value 追加到旧 value
strlen key # 返回字符串的长度


incrbyfloat key 3.5 # 增加 key 对应的值3.5
getrange key start end # 获取字符串指定下标所有的值
setrange key index value # 设置指定下标所有对应的值

以上除了 mset 和 mget 命令，时间复杂度都是O(1)

使用 mget 可以省去大量的网络时间

3.5 哈希表

哈希键值结构

常用命令

hget key field # 获取 hash key 对应的 field 的 value
hset key field value # 设置 hash key 对应 field 的 value
hdel key field # 删除 hash key 对应 field 的 value

hexists key field # 判断 hash key 是否有 field
hlen key # 获取 hash key field 的数量

hmget key field1 field2... # 获取一批 O(N)
hmset key field1 value1 field2 value2 # O(N)

hgetall key # 返回 hash key 对应所有的 field 和 value O(N)
hvals key # 返回所有的 value O(N)
hkeys key # 返回所有的 field O(N)

hsetnx key field value # 设置 hash key 对应 field 的 value (如果 field 存在，则失败)
hincrby key field intCounter # hash key 对应的 field 的 value 自增 intCounter
hibcrbyfloat key field floatCounter # hincrby 浮点数版

3.6 列表

结构

常用命令

rpush key value1 value2 ...valueN # 从列表右端插入值（1 - N 个）
lpush key value1 value2 ...valueN # 从列表左端插入值（1 - N 个）

linsert key before|after value newValue # 在list指定的值前｜后插入newValue O(N)

lpop key # 从左边弹出一个item
rpop key # 从右边弹出一个item

lrem key count value # 根据 count 值，从列表中删除所有 value 相等的项 O(N)
（1）count > 0，从左到右，删除最多 count 个 value 相等的项
（2）count < 0，从右到左，删除最多 Math.abs(count) 个 value 相等的项
（3）count = 0，删除所有 value 相等的项

ltrim key start end # 按照索引范围修剪列表 O(N)

lrange key start end # 获取列表指定索引范围所有 item O(N)
lindex key index # 获取列表指定索引的 item O(N)
llen key # 获取列表长度

lset key index newValue # 设置列表指定索引值为 newValue O(N)

---
blpop key timeout # lpop 阻塞版本，timeout 是阻塞超时时间，timeout=0 为永远不阻塞
brpop key timeout # 从右

sismember key value # 判断 value 是否在集合中

spop key # 从集合中随机弹出一个元素

sdiff key1 key2 # 差集

3.8 有序集合

结构

常用命令

zadd key score element(可以是多对) # 添加 score 和 element O(logN)
zrem key element(可以是多对) # 删除元素

zscore key element # 返回元素的分数
zincrby key increScore element # 增加或减少元素的分数

四、Redis 客户端的使用

4.1 Jedis 直连

# 1.生成一个 Jedis 对象，这个对象负责和指定 Redis 节点进行通信
Jedis jedis = new Jedis("127.0.0.1"，6379);
# 2.jedis 执行 set 操作
jedis.set("hello"，"world");
# 3.jedis 执行 get 操作，value = "world"
String value = jedis.get("hello")
  • host : Redis 节点的所在机器 IP
  • port : Redis 节点的端口
  • connection Timeout : 客户端连接超时
  • soTimeout : 客户端读写超时

五、瑞士军刀 Redis 其他功能

5.1 慢查询

生命周期

  • 慢查询发生在第三阶段
  • 客户端超时不一定慢查询，但慢查询是客户端超时的一个可能因素

两个配置

slowlog-max-len

  • 先进先出队列
  • 固定长度
  • 保存在内存内

slowlog-log-slower-than

  • 慢查询阈值（微秒）
  • slowlog-log-slower-than=0 记录所有命令（一般不会这么做）
配置方法

1.默认值

  • config get slowlog-max-len = 128
  • config get slowlog-log-slower-than = 10000

2.修改配置文件重启（不建议）

3.动态配置

  • config set slowlog-max-len 1000
  • config set slowlog-log-slower-than 1000

慢查询命令

  • slowlog get [n] : 获取慢查询队列 n-条数
  • slowlog len : 获取慢查询队列长度
  • slowlog reset : 清空慢查询队列

运维经验

  • slowlog-max-len 不要设置过大，默认 10ms，通常设置 1ms
  • slowlog-log-slower-than 不要设置过小，通常设置 1000 左右
  • 理解命令生命周期
  • 定期持久化慢查询

5.2 pipeline

什么是流水线

  • Redis 命令时间是微秒级别的
  • pipeline 每次条数要控制（网络）

pipeline-Jedis

maven 依赖

<dependency>
  	<groupId>redis.clients</groupId>
  	<artifactId>jedis</artifactId>
  	<version>2.9.0</version>
  	<type>jar</type>

使用 pipeline

mget、mset 是原子的

pipeline 命令会被拆分，是非原子的

使用建议

  • 注意每次 pipeline 携带数据量
  • pipeline 每次只能作用在一个 Redis 节点上
  • M 操作与 pipeline 的区别

5.3 发布订阅

角色

  • 发布者
  • 订阅者
  • 频道

和消息队列好像啊

模型

API

  • publish 发布
  • subscribe 订阅
  • unsubscribe 取消订阅
  • 其他
publish channel message # 发布命令
subscribe [channel] # 一个或多个
unsubscribe [channel] # 一个或多个
  • 消息队列是只有一个消费者能消费
  • 发布订阅是通知所有消费者

5.4 bitmap

Redis 可以直接对位操作

API

setbit key offset value # 给位图指定索引设置值
getbit key offset # 获取位图指定索引的值
bitcount key [start end] # 获取位图指定范围（start到end，单位为字节，不指定就是获取全部）位值为 1 的个数
bitop op destkey key [key...] # 做多个 Bitmap 的交集、并集、非、异或操作，并将结果保存在 destkey 中
bitpos key targetBit [start] [end] # 计算位图指定范围，第一个偏移量对应的值等于targetBit的位置

使用经验

  • type=string，最大 512MB
  • 注意 setbit 时的偏移量，可能有较大耗时
  • 位图不是绝对好

5.5 HyperLogLog

特性

  • 基于 HyperLogLog 算法：极小空间完成独立数量统计
  • 本质还是字符串

API

pfadd key element [element...] # 向 hyperloglog 添加元素
pfcount key [key...] # 计算 hyperloglog 的独立总数
pfmerge destkey sourcekey [sourcekey...] # 合并多个hyperloglog

使用经验

  • 是否能容忍错误？（错误率：0.81%）
  • 是否需要单条数据

5.6 geo

GEO 是什么

存储经纬度，计算两地距离，范围计算等

应用场景

  • 微信摇一摇（微信不一定用的redis，只是类似的功能）
  • 距离计算周围的酒店餐馆

API

geo key longitude latitude member # 增加地理位置信息
geopos key member [member...] # 获取地理位置信息
geodist key member1 member2 [unit] # 获取两个地理位置的距离 unit：单位

相关说明

  • redis3.2 版本之后提供
  • 使用 zset 实现的
  • 没有删除 API : zrem key member

六、Redis 持久化的取舍和选择

6.1 持久化的作用

什么是持久化

Redis 所有数据保持在内存中，对数据的更新将异步地保存到磁盘上。

持久化方式

  • 快照（RDB）
  • 写日志（AOP）

6.2 RDB

什么是 RDB

触发机制 - 主要三种方式

  • save（同步）
  • bgsave（异步）
  • 自动

save 命令

bgsave 命令

save 与 bgsave

自动生成 RDB

触发机制 - 不容忽略方式

  • 全量复制
  • debug reload
  • shutdown

RDB 总结

  • RDB 是 Redis 内存到硬盘的快照，用于持久化
  • save 通常会阻塞 Redis
  • bgsave 不会阻塞 Redis，但是会 fork 新进程
  • save 自动配置满足任意条件就会执行
  • 有些触发机制不容忽视
